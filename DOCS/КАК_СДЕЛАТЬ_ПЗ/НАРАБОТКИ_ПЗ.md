# 1 Введение

Современные компании всё чаще интегрируют генеративные нейросети в свои бизнес-процессы: автоматизация поддержки, анализ документов, генерация отчётов, помощь разработчикам. Каждая интеграция требует настройки подключения к LLM-провайдеру, управления историей диалогов, обработки потоковых ответов, контроля доступа и безопасности. Без единого сервиса каждая команда вынуждена реализовывать эту инфраструктуру заново. Система GoPengAI решает эту проблему, предоставляя готовый микросервис-шлюз с агентной архитектурой, древовидной историей и набором встроенных инструментов.

Под генеративными нейросетевыми моделями (LLM) понимается класс моделей машинного обучения, способных генерировать текстовый контент, отвечать на вопросы, анализировать данные и выполнять инструкции на естественном языке. Примерами таких моделей являются GPT-4, Claude, YandexGPT и GigaChat. Микросервисная архитектура, в свою очередь, предполагает построение приложения как набора небольших, слабо связанных сервисов, каждый из которых реализует одну бизнес-функцию. Под агентом (AI Agent) понимается программная сущность, которая использует LLM для планирования и выполнения действий: вызова инструментов, обращения к памяти, делегирования подзадач. Server-Sent Events (SSE) — это протокол потоковой передачи данных от сервера к клиенту по HTTP, используемый в проекте для стриминга ответов LLM в реальном времени.

GoPengAI — это микросервис, который принимает запросы через HTTP API или CLI, создаёт диалоговые сессии с LLM, сохраняет историю в виде дерева с возможностью ветвления (аналогично системе управления версиями git), предоставляет агентам инструменты (веб-поиск, сохранение и извлечение из памяти, делегирование под-агентам), конфигурируется через Markdown-файлы с YAML-заголовками и передаёт ответы потоково через SSE.

Разработка велась на языке Go версии 1.22+ с использованием следующих инструментов: встроенный пакет net/http с новой маршрутизацией для HTTP-сервера, Cobra для CLI-интерфейса, SQLite (библиотека ncruces/go-sqlite3 без CGo, через WASM) в качестве встраиваемой базы данных, sqlc для генерации типобезопасного Go-кода из SQL-запросов и Goose для управления миграциями базы данных.

# 2 Постановка задачи

Целью данной работы является проектирование и разработка микросервиса для интеграции генеративных нейросетевых моделей в корпоративную инфраструктуру. Система должна предоставлять унифицированный интерфейс для взаимодействия с LLM-провайдерами, управлять историей диалогов и обеспечивать возможность расширения функциональности через подключаемые инструменты.

Система GoPengAI предоставляет следующие функциональные возможности. Для пользователей (клиентов микросервиса): управление сессиями (создание, просмотр, удаление диалоговых сессий, переключение между сессиями), отправка сообщений (отправка сообщения в сессию с получением ответа от LLM в режиме реального времени через SSE), работа с агентами (выбор агента для диалога, просмотр списка доступных агентов), ветвление истории (редактирование отправленного сообщения с созданием новой ветки без потери исходной истории), использование инструментов (агент может выполнять веб-запросы, сохранять и извлекать данные из памяти, делегировать подзадачи другим агентам).

Интерфейсы взаимодействия: HTTP API — RESTful-интерфейс для интеграции с внешними системами, совместимый с OpenAI API-форматом; CLI (Command Line Interface) — утилита командной строки для локального использования, реализованная через Cobra.

Система построена на принципах событийно-ориентированной архитектуры (Event-Driven Architecture) с разделением на три логических слоя. Слой инфраструктуры (API, CLI, БД, конфигурация) отвечает за взаимодействие с внешним миром и хранение данных. Слой бизнес-логики (агентный движок, история, инструменты, LLM-клиент) реализует интеллектуальную обработку запросов. Слой конфигурации (Markdown-файлы агентов, JSON-конфиг) определяет поведение системы без изменения кода.

# 3 Анализ требований к системе

Требования к серверной части. Минимальные требования: 64-битный процессор, 2 ядра, 2.0 ГГц; 1 ГБ оперативной памяти (без локальной LLM) или 4 ГБ (с локальной LLM); 100 МБ дискового пространства (бинарник и БД); операционная система Linux (любой дистрибутив), macOS 12+ или Windows 10+. Рекомендуемые требования: 4+ ядер, 2.5+ ГГц; 4 ГБ ОЗУ (без LLM) или 16 ГБ (с локальной LLM); 500 МБ дискового пространства.

Требования к программному окружению. Для сборки из исходников требуется Go версии 1.22 или выше. SQLite встраивается в бинарник через WASM-прокси, отдельная установка не требуется. Для работы необходим LLM-провайдер — любой сервис с OpenAI-совместимым API: OpenAI, Anthropic через proxy, YandexGPT, GigaChat, локальные модели через Ollama или vLLM.

Требования к сети. Для работы агента требуется доступ к LLM-провайдеру по HTTPS. Для локального использования возможна работа без доступа в интернет при использовании локальной LLM через Ollama. CLI-клиент должен иметь доступ к API-серверу по HTTP или HTTPS.

Дополнительные требования. Безопасность: аутентификация запросов через Bearer-токен. Надёжность: восстановление после паник (recovery middleware), graceful shutdown. Производительность: потоковая передача ответов через SSE без блокировки соединения.

# 4 Теоретическое описание решения задачи

## 4.1 Событийно-ориентированная архитектура (Event-Driven Architecture)

В основе коммуникации между компонентами системы лежит EventBus — шина событий в памяти, работающая по принципу «издатель-подписчик» (pub/sub). Движок агента публикует события: message_sent, tool_called, error, done. HTTP-обработчики подписываются на события и транслируют их клиенту через SSE. Каждый компонент остаётся независимым: движок не знает, кто слушает его события.

При традиционном синхронном подходе клиент ждал бы полного ответа LLM, который может длиться от 10 до 30 секунд. EventBus в сочетании с SSE позволяет передавать ответ по мере генерации: пользователь видит текст в реальном времени.

## 4.2 Древовидная история сообщений (Tree-based History)

В отличие от традиционных чатов с линейной историей, GoPengAI хранит сообщения в виде дерева. Каждое сообщение имеет поле parent_id, указывающее на предыдущее сообщение. При редактировании сообщения создаётся новая ветка: исходное сообщение не теряется. Сессия может содержать несколько веток, аналогично git-репозиторию.

Для построения полного контекста используется рекурсивный CTE (Common Table Expression) в SQL. Движок агента выбирает активную ветку и собирает все сообщения от корня до листа.

## 4.3 Паттерн «Агент-Инструмент» (Agent-Tool Pattern)

GoPengAI реализует классический паттерн AI-агентов, состоящий из следующих этапов. Планирование: LLM получает контекст и список доступных инструментов. Вызов инструмента: LLM решает, какой инструмент вызвать, и формирует JSON-аргументы. Исполнение: движок выполняет инструмент и возвращает результат LLM. Цикл: LLM анализирует результат и либо вызывает следующий инструмент, либо формирует финальный ответ.

Реализованные инструменты: web_fetch — HTTP-запрос к внешнему URL с извлечением текста; memory_save и memory_recall — сохранение и извлечение данных из привязанной к агенту памяти; delegate — делегирование подзадачи другому агенту.

## 4.4 Типобезопасные SQL-запросы (sqlc)

Вместо традиционного ORM (например, GORM) в проекте используется sqlc — генератор Go-кода из SQL-запросов. Разработчик пишет чистый SQL, получая полный контроль над запросами. sqlc генерирует типобезопасные Go-функции: ошибки типов выявляются на этапе компиляции. Отсутствие runtime-магии ORM обеспечивает предсказуемую производительность.

## 4.5 Управление миграциями (Goose)

Goose — инструмент для версионирования схемы базы данных. Каждая миграция представляет собой SQL-файл с версией (например, 001_init.sql). При запуске сервера неприменённые миграции выполняются автоматически.

# 5 Архитектура системы

## 5.1 Общая архитектура

Система GoPengAI построена по модульному принципу и включает следующие компоненты.

Точки входа: cmd/api/main.go и cmd/cli/main.go (разработчик Гамидов). Слой HTTP-обработки (internal/api/): обработчики запросов (handler.go), маршрутизация (routes.go), события и SSE-стриминг (events.go), middleware для логирования, CORS, recovery и аутентификации (middleware.go). Слой движка агента (internal/agent/): основной цикл агента (engine.go), загрузчик конфигурации агентов из Markdown-файлов (loader.go), реестр агентов (registry.go). Слой истории (internal/history/): древовидная структура сообщений (tree.go), репозиторий для работы с БД (repo.go), управление ветками (branch.go), сборка контекста для LLM (context.go). Слой инструментов (internal/tools/): реестр инструментов (registry.go), веб-поиск (web_fetch.go), память агента (memory.go), делегирование (delegate.go). Слой LLM-клиента (internal/llm/): HTTP-клиент для OpenAI-совместимых API (client.go), типы данных (types.go), потоковая обработка ответов (stream.go). Слой базы данных (internal/db/): подключение к SQLite (connect.go), миграции через Goose (migrations/), сгенерированный sqlc-код (.sql и .go файлы). Слой конфигурации (internal/config/): структура и загрузка конфигурации (config.go), поддержка JSON-файла и переменных окружения. Конфигурация агентов (agents/): Markdown-файлы с YAML frontmatter, определяющие поведение агентов без изменения кода.

## 5.2 Поток данных

Обработка запроса в системе происходит следующим образом. Пользователь отправляет HTTP-запрос на endpoint /api/chat. HTTP-обработчик принимает запрос и вызывает метод Process движка агента. Движок агента строит контекст, извлекая историю сообщений из базы данных через рекурсивный CTE. Движок отправляет запрос к LLM-провайдеру через LLM-клиент. Ответ LLM передаётся потоково через SSE-канал обратно пользователю. Если LLM решает вызвать инструмент, движок выполняет соответствующий инструмент и передаёт результат обратно LLM для продолжения цикла. По завершении цикла новое сообщение сохраняется в базу данных.

## 5.3 Схема базы данных

База данных SQLite содержит пять таблиц.

Таблица sessions предназначена для хранения диалоговых сессий. Ключевые поля: id, name, agent_id, created_at.

Таблица messages хранит сообщения в древовидной структуре. Ключевые поля: id, session_id, parent_id, role, content, created_at. Поле parent_id содержит NULL для корневого сообщения. Рекурсивный CTE обходит дерево для сборки контекста.

Таблица branches отслеживает активные ветки сессий. Ключевые поля: session_id, head_message_id.

Таблица configs предназначена для хранения конфигураций. Ключевые поля: id, key, value.

Таблица memory хранит память агентов. Ключевые поля: id, agent_id, key, value, created_at.

# 7 Тестирование продукта

## 7.1 Модульное тестирование

Модульное тестирование выполнено с использованием стандартного пакета testing языка Go. Тесты покрывают следующие компоненты.

Тестирование построения дерева истории (internal/history/). Проверяется корректность создания дерева сообщений, добавление новых сообщений с parent_id, создание новых веток при редактировании, сборка контекста от корня к листу.

Тестирование загрузчика агентов (internal/agent/). Проверяется парсинг Markdown-файлов с YAML frontmatter, корректность извлечения системного промпта, списка инструментов и параметров модели.

Тестирование LLM-клиента (internal/llm/). Используется мок-сервер (httptest) для эмуляции OpenAI-совместимого API. Проверяется формирование запроса, обработка потокового ответа, обработка ошибок.

Тестирование HTTP-обработчиков (internal/api/). Используется httptest для эмуляции HTTP-запросов. Проверяются все endpoints: создание сессии, отправка сообщения, просмотр истории, удаление сессии.

Тестирование инструментов (internal/tools/). Проверяется корректность веб-запроса (web_fetch), сохранение и извлечение данных из памяти (memory), делегирование подзадачи (delegate).

# 8 Описание программы

## 8.1 Руководство программиста

### Системные требования

- Go версии 1.22 или выше
- Make (опционально, для использования Makefile)
- Доступ к LLM-провайдеру (OpenAI, YandexGPT и т.д.)

### Сборка и запуск

Клонировать репозиторий: git clone <url> и перейти в директорию gopen. Установить зависимости командой go mod download. Собрать проект: make build. Создать файл конфигурации gopengai.json на основе gopengai.json.example, указав API-ключ LLM-провайдера. Запустить сервер: ./gopen api. Использовать CLI: ./gopen chat --session "Моя сессия".

### Конфигурация

Файл gopengai.json содержит следующие секции. Секция llm: provider (провайдер), model (модель), api_key (ключ API), base_url (базовый URL). Секция db: path (путь к файлу базы данных). Секция server: addr (адрес сервера), auth_token (токен аутентификации). Все поля могут быть переопределены через переменные окружения.

### Миграции базы данных

Миграции выполняются автоматически при запуске сервера. Для ручного управления доступны команды: ./gopen db migrate (применить миграции), ./gopen db rollback (откатить последнюю миграцию).

## 8.2 Руководство пользователя

### CLI-интерфейс

Таблица 4 — Команды CLI-интерфейса

| Команда | Описание | Пример |
|---|---|---|
| gopen chat | Быстрый чат (один запрос) | gopen chat --msg "Привет" |
| gopen chat (REPL) | Интерактивный режим | gopen chat --verbose |
| gopen session list | Список сессий | gopen session list |
| gopen session create --name X | Создать сессию | gopen session create --name "Работа" |
| gopen session show --id X | Показать историю | gopen session show --id abc123 |
| gopen session delete --id X | Удалить сессию | gopen session delete --id abc123 |

### HTTP API

HTTP API совместим с форматом OpenAI API и предоставляет следующие endpoint'ы.

POST /api/chat — отправка сообщения в сессию. GET /api/sessions — получение списка сессий. POST /api/sessions — создание новой сессии. GET /api/sessions/{id} — просмотр сессии. DELETE /api/sessions/{id} — удаление сессии.

Пример запроса:

curl -X POST http://localhost:8080/api/chat \
  -H "Authorization: Bearer my-secret-token" \
  -H "Content-Type: application/json" \
  -d '{
    "session_id": "abc123",
    "message": "Расскажи про Go"
  }'

# Заключение

В ходе выполнения работы спроектирован и реализован микросервис GoPengAI для интеграции генеративных нейросетевых моделей в корпоративную инфраструктуру. Реализована агентная архитектура с древовидной историей сообщений, поддерживающая ветвление диалогов. Система предоставляет два интерфейса взаимодействия: CLI и HTTP API, совместимый с форматом OpenAI.

В разработке использованы современные технологии: Go версии 1.22+, встраиваемая база данных SQLite (через WASM без CGo), sqlc для генерации типобезопасного кода из SQL-запросов, Goose для управления миграциями, Cobra для CLI-интерфейса и Server-Sent Events для потоковой передачи ответов в реальном времени.

Распределение обязанностей между двумя разработчиками выполнено по модульному принципу: Тимофеев Илья отвечал за бизнес-логику (движок агента, история, инструменты, LLM-клиент), Гамидов Адиль — за инфраструктуру (HTTP-сервер, CLI, база данных, конфигурация).

Система готова к дальнейшему расширению: добавление новых инструментов, поддержка дополнительных LLM-провайдеров, реализация новых транспортных протоколов.
