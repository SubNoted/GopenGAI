# База данных SQLite (internal/db)

## Общий подход

Весь слой базы данных построен на трёх библиотеках, работающих в связке:

```
SQL-миграции (Goose) → Схема БД → SQL-запросы → sqlc → Типобезопасный Go-код
```

## 1. Подключение к SQLite (connect.go)

### Как открывается соединение

```go
db, err := sql.Open("sqlite3", path)
```

Драйвер `ncruces/go-sqlite3` регистрируется через `_ "github.com/ncruces/go-sqlite3/driver"`.

### Pragmas (настройки SQLite)

Сразу после открытия применяются:

| Pragma | Значение | Зачем |
|--------|----------|-------|
| `foreign_keys` | ON | Проверка внешних ключей |
| `journal_mode` | WAL | Параллельные чтения + запись без блокировок |
| `page_size` | 4096 | Стандартный размер страницы |
| `cache_size` | -8000 | Кэш ~8 МБ (отрицательное = килобайты) |
| `synchronous` | NORMAL | Баланс безопасности и скорости (в WAL достаточно) |
| `busy_timeout` | 5000 | Ждать до 5 секунд при блокировке |

### Pooling-настройки

```go
db.SetMaxOpenConns(1)    // SQLite — однопоточный писатель
db.SetMaxIdleConns(1)    // Минимум простаивающих соединений
db.SetConnMaxLifetime(time.Hour)  // Переоткрытие соединения раз в час
```

> **Почему MaxOpenConns=1?** SQLite не поддерживает конкурентную запись из нескольких соединений. Даже с WAL режимом, пишет только одно соединение. Ограничение в 1 соединение гарантирует отсутствие `database is locked` ошибок.

## 2. Миграции (embed.go + Goose)

### Как работают миграции

1. **SQL-файлы** лежат в `internal/db/migrations/`
2. Встраиваются в бинарник через `//go:embed migrations/*.sql`
3. Goose запускает `Up()` при старте сервера (в `cmd/api/main.go`)

### Миграция 001: пять таблиц

#### sessions — Сессии (диалоги)

```sql
CREATE TABLE sessions (
    id TEXT PRIMARY KEY,
    parent_session_id TEXT,       -- для форков сессий
    agent_name TEXT NOT NULL DEFAULT 'default',
    title TEXT NOT NULL,
    active_leaf_id TEXT,          -- текущая активная ветка
    status TEXT NOT NULL DEFAULT 'idle',  -- idle | working | error
    message_count INTEGER NOT NULL DEFAULT 0,
    prompt_tokens INTEGER NOT NULL DEFAULT 0,
    completion_tokens INTEGER NOT NULL DEFAULT 0,
    cost REAL NOT NULL DEFAULT 0.0,
    updated_at INTEGER NOT NULL,
    created_at INTEGER NOT NULL,
    summary_message_id TEXT
);
```

#### messages — Сообщения (древовидные)

```sql
CREATE TABLE messages (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    parent_id TEXT,              -- родительское сообщение (для дерева)
    role TEXT NOT NULL,          -- user | assistant | tool | system
    parts TEXT NOT NULL DEFAULT '[]',
    content TEXT,
    agent_name TEXT,
    tool_name TEXT,
    tool_call_id TEXT,
    tool_args TEXT,
    model TEXT,
    token_count INTEGER DEFAULT 0,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    finished_at INTEGER,
    FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);
```

> **Ключевая идея:** `parent_id` создаёт связный список, который через рекурсивный CTE можно развернуть в дерево. Это позволяет ветвить диалоги.

#### agents — Кеш агентов (загруженных из .md файлов)

```sql
CREATE TABLE agents (
    name TEXT PRIMARY KEY,
    system_prompt TEXT NOT NULL,
    tools TEXT NOT NULL DEFAULT '[]',
    model TEXT,
    parent_agent TEXT,
    permissions TEXT NOT NULL DEFAULT '{}',
    config_path TEXT,
    loaded_at INTEGER NOT NULL
);
```

#### memory — Долговременная память агентов

```sql
CREATE TABLE memory (
    id TEXT PRIMARY KEY,
    agent_name TEXT NOT NULL,
    key TEXT NOT NULL,
    value TEXT NOT NULL,
    category TEXT,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    FOREIGN KEY (agent_name) REFERENCES agents(name) ON DELETE CASCADE
);
```

#### delegation_logs — Логи делегирования

```sql
CREATE TABLE delegation_logs (
    id TEXT PRIMARY KEY,
    parent_message_id TEXT NOT NULL,
    child_agent_name TEXT NOT NULL,
    child_session_id TEXT,
    task_description TEXT NOT NULL,
    result_summary TEXT,
    duration_ms INTEGER,
    created_at INTEGER NOT NULL,
    FOREIGN KEY (parent_message_id) REFERENCES messages(id) ON DELETE CASCADE,
    FOREIGN KEY (child_agent_name) REFERENCES agents(name) ON DELETE CASCADE
);
```

### Триггеры

| Триггер | Действие |
|---------|----------|
| `update_sessions_updated_at` | При UPDATE sessions → обновить `updated_at` |
| `update_messages_updated_at` | При UPDATE messages → обновить `updated_at` |
| `update_memory_updated_at` | При UPDATE memory → обновить `updated_at` |
| `update_session_message_count_on_insert` | При INSERT в messages → увеличить `message_count` в sessions |
| `update_session_message_count_on_delete` | При DELETE из messages → уменьшить `message_count` |

### Индексы

```sql
CREATE INDEX idx_messages_session_id ON messages (session_id);
CREATE INDEX idx_messages_parent_id ON messages (parent_id);
CREATE INDEX idx_memory_agent_name ON memory (agent_name);
CREATE INDEX idx_delegation_logs_parent ON delegation_logs (parent_message_id);
```

## 3. sqlc (type-safe query generation)

### Конфигурация (sqlc.yaml)

```yaml
version: "2"
sql:
  - engine: "sqlite"
    schema: "internal/db/migrations"
    queries: "internal/db/sql"
    gen:
      go:
        package: "db"
        out: "internal/db"
        emit_json_tags: true
        emit_prepared_queries: true
        emit_interface: true
```

### Что генерирует sqlc

| Файл | Содержимое |
|------|-----------|
| `db.go` | `Queries` struct с методами для всех запросов |
| `models.go` | Go-структуры для каждой таблицы (`Session`, `Message`, `Agent`, `Memory`, `DelegationLog`) |
| `querier.go` | Интерфейс `Querier` — все методы |
| `sessions.sql.go` | Реализация CRUD для сессий |
| `messages.sql.go` | Реализация CRUD для сообщений + рекурсивный CTE |
| `agents.sql.go` | CRUD для агентов |
| `memory.sql.go` | CRUD для памяти |
| `delegation_logs.sql.go` | CRUD для логов делегирования |

### SQL-запросы (пример messages.sql)

```sql
-- name: CreateMessage :one
INSERT INTO messages (...) VALUES (...) RETURNING *;

-- name: GetBranchFromRootTo :many
WITH RECURSIVE branch AS (
    SELECT * FROM messages WHERE messages.id = ?
    UNION ALL
    SELECT m.* FROM messages m
    INNER JOIN branch b ON m.id = b.parent_id
)
SELECT * FROM branch ORDER BY created_at ASC;

-- name: GetAllLeaves :many
SELECT m.* FROM messages m
WHERE m.session_id = ?
AND m.id NOT IN (
    SELECT sub.parent_id FROM messages sub
    WHERE sub.parent_id IS NOT NULL AND sub.session_id = ?
);
```

> **Рекурсивный CTE** — ключевая фича для древовидной истории. Позволяет за один SQL-запрос получить всю ветку от любого сообщения до корня.

### Как использовать сгенерированный код

```go
// Создаём Queries
q := db.New(sqlDB)

// Создаём сессию
session, err := q.CreateSession(ctx, db.CreateSessionParams{
    ID:        newID(),
    AgentName: "default",
    Title:     "Chat",
    Status:    "idle",
    CreatedAt: time.Now().UnixMilli(),
    UpdatedAt: time.Now().UnixMilli(),
})

// Получаем ветку сообщений
msgs, err := q.GetBranchFromRootTo(ctx, leafMessageID)
```

### Транзакции

sqlc поддерживает транзакции через `WithTx`:

```go
tx, err := sqlDB.BeginTx(ctx, nil)
qtx := q.WithTx(tx)

// Все операции в рамках одной транзакции
qtx.CreateMessage(ctx, ...)
qtx.UpdateSession(ctx, ...)

tx.Commit() // или tx.Rollback()
```

Это критически важно для паттерна **read-modify-write** в хендлере чата, чтобы избежать race condition.
