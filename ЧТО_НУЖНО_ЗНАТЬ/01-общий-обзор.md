# GoPengAI — Общий обзор проекта

## Что это такое?

**GoPengAI** — это AI-агентная система, написанная на Go (Golang). Она позволяет:
- Создавать "сессии" — изолированные диалоги с LLM
- Отправлять сообщения и получать ответы от OpenAI-совместимых моделей
- Сохранять историю диалогов в SQLite с поддержкой **древовидной структуры** (ветвление, форки, как в git)
- Использовать **инструменты** (tools): веб-поиск, сохранение/чтение памяти, делегирование под-агентам
- Конфигурировать **агентов** через Markdown-файлы с YAML-заголовками
- Потоково передавать ответы через **SSE (Server-Sent Events)**

## Как это работает (архитектурно)

```
Клиент (curl / CLI)
       │
       ▼
  HTTP API (net/http)
       │
       ├──→ POST /session/{id}/message (sync MVP)
       │       │
       │       ├──→ SQLite (сохраняем сообщение + триггеры)
       │       │       │
       │       │       └──→ Рекурсивный CTE для ветки истории
       │       │
       │       ├──→ LLM Client → OpenAI API (c поддержкой tool calling)
       │       │
       │       ├──→ SQLite (сохраняем ответ ассистента)
       │       │
       │       ├──→ JSON-ответ клиенту
       │       │
       │       └──→ Загрузчик агентов из .md (Agent types + Registry — ✅)
       │
       ├──→ POST /v1/chat/completions (OpenAI pass-through)
       │
       └──→ GET /health

Планируется (не реализовано):
       Agent Engine (горутина) → EventBus → SSE → Client
       Tools (web_fetch | memory | delegate)
       Асинхронный POST → 202 + SSE streaming
```

## Технический стек

| Компонент | Технология | Причина выбора |
|-----------|-----------|----------------|
| Язык | Go 1.21+ | Один бинарник, никаких зависимостей на рантайм |
| HTTP | `net/http` (stdlib) | Новый роутер Go 1.22+ с path params |
| База данных | SQLite | Никакого отдельного сервера, один .db файл на проект |
| Драйвер SQLite | `ncruces/go-sqlite3` | **Чистый Go** — компилируется через WASM, не нужен CGo |
| Миграции | Goose (`pressly/goose/v3`) | SQL-миграции, встроенные в бинарник |
| Генерация Go-кода SQL | sqlc | Типобезопасные запросы, никаких ORM |
| CLI | Cobra (`spf13/cobra`) | Стандарт де-факто для Go CLI |
| Формат агентов | Markdown + YAML frontmatter | Агенты как документы, не как код |

## Ключевые архитектурные решения (ADR)

### 1. Чистый Go без CGo
`ncruces/go-sqlite3` использует WebAssembly вместо C-вызовов. Это значит:
- Кросскомпиляция без мучений (не нужен gcc/musl)
- Один бинарник для любой платформы
- Возможный риск производительности (Wasm overhead) — пока не выявлен

### 2. SQLite в режиме WAL
- `journal_mode=WAL` — конкурентные чтения и запись
- `SetMaxOpenConns(1)` — SQLite однопоточный писатель
- `busy_timeout=5000` — ждать до 5 секунд при блокировке
- `synchronous=NORMAL` — баланс скорости и безопасности

### 3. Транзакции для безопасности
Критический паттерн — **read-modify-write** в рамках одной транзакции:
- Читаем сессию
- Создаём сообщение пользователя
- Обновляем статус сессии
- Фиксируем транзакцию
- Затем — долгий LLM-вызов (уже без транзакции)
- После LLM — новая транзакция для сохранения ответа

Это предотвращает race condition, когда два запроса к одной сессии создадут «осиротевшие» сообщения.

### 4. ID без зависимостей
Вместо библиотеки `github.com/google/uuid` — собственная функция `newID()`:
- 16 байт из `crypto/rand`
- Формат: `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`
- **Fallback**: если `crypto/rand` отказал (крайне редко), паника заменена на timestamp-based ID с логгированием

### 5. Безопасность ошибок
Реальные ошибки (SQL schema, file paths) пишутся в server-side log, клиенту возвращается `"internal server error"`. Исключение — явные ошибки пользователя (404, 400).

### 6. Rune-безопасное обрезание строк
Обрезание строк в CLI считает руны, а не байты, чтобы не сломать многобайтовые символы (кириллица, эмодзи).
