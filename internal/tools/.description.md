### `internal/tools`
Tool implementations directory. Phase 5 complete.

### `registry.go`
Tool interface + concurrency-safe registry + permission checking.

#### Algorithm
- `Tool` interface: `Name()`, `Description()`, `Parameters()`, `Execute(ctx, args)`
- `Registry` struct with `sync.RWMutex`-guarded `map[string]Tool`
- `Register(tool)`, `Get(name)`, `List()`, `Size()` — standard registry ops
- `ToToolDefinitions()` — converts all registered tools to LLM API format (`[]ToolDefinition`)
- `IsAllowed(toolName, permissions)` — checks agent permission map (default-allow if empty, default-deny for explicit maps)
- Local `ToolDefinition`/`ToolFunction` types mirror `llm` package to avoid circular imports

#### Data
- **Flow:** NewRegistry → Register(tool) → Get/List/ToToolDefinitions → use in agent engine

### `web_fetch.go`
WebFetchTool — fetches URL, strips HTML, returns text content.

#### Algorithm
1. Parse JSON args `{ "url": "..." }`
2. Validate URL scheme (reject non-http/https — SSRF prevention)
3. HTTP GET with User-Agent `GoPengAI/1.0`, 10s timeout, redirect policy (reject cross-scheme redirects)
4. Read up to 50KB response body
5. Strip HTML: (a) remove `<script>`/`<style>` blocks with content, (b) strip remaining tags, (c) decode entities, (d) collapse whitespace
6. Truncate to ~10000 display chars

#### Data
- **Structures:** `WebFetchTool{ Client, MaxResponseBytes }`
- **Flow:** Execute → HTTP GET → read → stripHTML → return text

### `memory.go`
MemorySave + MemoryRecall tools — persistent agent-scoped key-value storage.

#### Algorithm
**MemorySave:**
1. Parse args `{ "key", "value", "category?" }`
2. Extract `agent_name` from context (ContextKeyAgentName)
3. Get `db.Querier` from context (ContextKeyQuerier)
4. Call `db.CreateMemory()` with new ID + timestamps
5. Return confirmation string

**MemoryRecall:**
1. Parse args `{ "key"? }`
2. If key provided: call `db.GetMemory(agentName, key)` → return single fact
3. If key empty: call `db.ListMemoryByAgent(agentName)` → return all facts as formatted text
4. Return formatted result or "none found" message

#### Data
- **Context keys:** `ContextKeyAgentName`, `ContextKeyQuerier`
- **Flow:** Execute → read context → db call → return formatted result

### `delegate.go`
DelegateTool — delegates a task to a sub-agent with cycle detection + timeout.

#### Algorithm
1. Parse args `{ "agent_name", "task" }`
2. **Cycle detection:** check `visited_agents` map from context — reject if agent already in chain; add current agent to visited set and propagate to child context
3. Load sub-agent from registry via `AgentRegistryLite` interface (avoids circular import)
4. Get `LLMClientLite` from context for the actual LLM call
5. Build messages: `[{ role: "system", content: subAgent.SystemPrompt }, { role: "user", content: task }]`
6. Call LLM with 30s timeout context
7. Log delegation to `delegation_logs` table via `db.CreateDelegationLog()`
8. Return formatted result string

#### Data
- **Context keys:** `ContextKeyAgentRegistry`, `ContextKeyLLMClient`, `ContextKeyVisitedAgents`, `ContextKeyCurrentMessageID`
- **Interfaces:** `AgentLite` (Name, SystemPrompt, Model, HasTool, IsToolAllowed), `AgentRegistryLite` (Get), `LLMClientLite` (ChatCompletion)
- **Flow:** Execute → cycle check → load sub-agent → LLM call → log → return result

### Context key conventions
Tools access dependencies via context values:
| Key | Type | Used By |
|-----|------|---------|
| `agent_name` | `string` | memory, delegate |
| `db_querier` | `db.Querier` | memory, delegate |
| `agent_registry` | `AgentRegistryLite` | delegate |
| `llm_client` | `LLMClientLite` | delegate |
| `visited_agents` | `map[string]bool` | delegate |
| `current_message_id` | `string` | delegate |

The agent engine (Phase 6) is responsible for populating these context values before calling tool Execute.
