Directory containing agent-related logic and structure.

### `engine.go`
Agent engine — core processing loop with tool execution, event publishing, abort support.

#### Algorithm
**Engine struct** holds: LLM client, tool registry, history repository, agent registry, DB, config, EventBus.

**Process(ctx, sessionID, userContent, agentName) error** — async entry point:
1. Create abort-capable context, store cancel func
2. Publish `session.status=working` (deferred: `session.status=idle`)
3. Load agent from registry, resolve model
4. Load session, save user message (parent = current active_leaf), track `parentMsgID`
5. Build LLM context: system prompt + active branch
6. Core loop (max N iterations from config):
   a. Call LLM with tool definitions
   b. If `stop` — save assistant message (parent = `parentMsgID`), update active leaf, publish `message.complete`, return
   c. If `tool_calls` — save assistant message (parent = `parentMsgID`, preserves Content alongside ToolCalls), set `parentMsgID = assistantMsg.ID`:
      - For each tool:
        - Publish `message.tool.started`
        - Check agent permission (`IsToolAllowed`)
        - If denied: save "tool denied" result, continue
        - Execute tool with context (agent name, querier, registry, LLM client)
        - Publish `message.tool.completed` or `message.tool.error`
        - Save tool result message (role=tool, parent = assistantMsg.ID)
        - Append to in-memory context
      - Continue loop (next LLM call chains from the tool-assistant message)
   d. If `length` — publish error, return
7. On error: publish `message.error`
8. Max iterations reached: publish error + return

**Tree integrity**: `parentMsgID` variable tracks the current branch tip through the loop. Starts as `userMsgID`, updated to each assistant message ID on tool_calls iterations. This ensures multi-turn tool use produces a proper tree chain (user → assistant(tool_calls) → tool results → assistant(stop)), not flat siblings.

**UpdateActiveLeaf ordering**: DB update happens BEFORE publishing `message.complete`, so UI never sees success for a failed DB write.

**Abort support:**
- `Abort(sessionID)` — cancels running process for session
- Context cancellation checked at loop boundaries and between tool calls
- Uses `sync.RWMutex`-guarded map of `sessionID → CancelFunc`

#### Adapters (avoid circular imports)
- `llmClientLite` — adapts `*llm.Client` to `tools.LLMClientLite`
- `agentRegistryLite` — adapts `*agent.Registry` to `tools.AgentRegistryLite`
- `agentLite` — adapts `*agent.Agent` to `tools.AgentLite`

#### Interfaces
- `EventBus` — `PublishGlobal(eventType, properties)`, `PublishSession(sessionID, eventType, properties)` — primitive types to avoid type coupling with `api` package
- `HistoryRepository` — subset of `history.Repository` (InsertMessage, GetSession, BuildContext, UpdateActiveLeaf) — avoids circular import

#### ID Generation
- `newID()` — 3 retries of `crypto/rand`, fallback to `engine-{timestamp}-{atomic counter}` to prevent collisions with concurrent engine calls

### `loader.go`
Loads agent configurations from `.md` files with YAML frontmatter.

#### Exports
- `LoadAgent(path string) (*Agent, error)` — Loads a single agent from a `.md` file.
- `LoadDirectory(dir string) (map[string]*Agent, error)` — Scans a directory for `.md` files and loads each as an Agent.

#### External Dependencies
- `gopkg.in/yaml.v3`

---

### `registry.go`
In-memory, concurrency-safe store for agent configurations.

#### Exports
- `NewRegistry() *Registry` — Creates an empty agent registry.

#### `Registry` Methods
- `Register(agent *Agent)` — Adds an agent to the registry (overwrites if exists).
- `Get(name string) (*Agent, error)` — Retrieves an agent by name.
- `List() []Agent` — Returns a copy of all registered agents.
- `Names() []string` — Returns the names of all registered agents.
- `Has(name string) bool` — Returns true if an agent with the given name is registered.
- `InitializeFromDir(dir string) (int, error)` — Loads all agents from a directory and registers them.
- `Size() int` — Returns the number of registered agents.

---

### `types.go`
Core data structures for agent representation and message handling.

#### Exports
- `Agent` struct — In-memory representation of an agent.
  - `HasTool(name string) bool` — Returns true if the agent has the given tool in its Tools list.
  - `IsToolAllowed(name string) bool` — Returns true if the tool is explicitly allowed (default deny).
- `Message` struct — Represents a single message within the agent engine.
- `ToolCall` struct — Represents a function call requested by the LLM.
- `Response` struct — Represents the final result from an agent processing cycle.
- `Usage` struct — Tracks token consumption for a response.
- `frontmatter` struct (internal) — YAML structure parsed from agent `.md` files.
